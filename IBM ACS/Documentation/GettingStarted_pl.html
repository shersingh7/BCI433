1.0 Wprowadzenie

Produkt IBM i Access Client Solutions to najnowszy element rodziny produktów IBM i Access. Udostępnia on oparty na języku Java i niezależny od platformy interfejs, który działa w większości systemów operacyjnych obsługujących środowisko Java, w tym w systemach Linux, Mac i Windows. Produkt IBM i Access Client Solutions konsoliduje w jednym uproszczonym miejscu najczęściej używane zadania dotyczące zarządzania systemem IBM i.

Produkt IBM i Access Client Solutions używa tych samych serwerów hostów IBM i co inne produkty rodziny IBM i Access oraz wymaga tej samej licencji rodziny IBM i Access (XW1) pozwalającej na korzystanie z funkcji emulacji terminala 5250 i przesyłania danych.

Dostępne są także dwa opcjonalne pakiety zawierające oprogramowanie pośrednie pozwalające na używanie i projektowanie aplikacji klienckich dla systemów Windows i Linux:
    IBM i Access Client Solutions – pakiet aplikacji systemu Windows
    IBM i Access Client Solutions – pakiet aplikacji systemu Linux

Klienci używający systemu IBM i w wersji 7.1 lub nowszej, którzy mają aktualne uprawnienia, mogą pozyskać produkt IBM i Access Client Solutions przy użyciu jednej z następujących metod:

  • Pobranie podstawowego pakietu IBM i Access Client Solutions.
  • Pobranie podstawowego pakietu i opcjonalnych pakietów aplikacji dla systemów Windows i Linux z serwisu WWW Entitled Software Support (ESS) z kluczem licencyjnym 5761-SS1 lub 5770-SS1.

Opcjonalne pakiety aplikacji dla systemów Windows i Linux są dostępne tylko w serwisie WWW Entitled Software Support (ESS).

Klienci mogą uzyskać nośniki, zamawiając klucz licencyjny 5761-SS1 lub 5770-SS1 (opcja odświeżenia 6288). Nośniki fizyczne zawierają możliwą do uruchomienia wersję produktu, która pozwala na uruchomienie produktu bezpośrednio z dysku CD. Nośniki fizyczne zawierają także archiwum zip produktu, które może zostać skopiowane i rozpakowane w wybranym położeniu. Nośniki fizyczne produktu IBM i Access Client Solutions nie zawierają opcjonalnych pakietów aplikacji systemu Windows i Linux.

Informacje dodatkowe można znaleźć pod adresem:

2.0 Funkcje

Produkt IBM i Access Client Solutions udostępnia następujące funkcje:

  • W pełni funkcjonalny emulator terminala 5250 oparty na produkcie IBM Rational Host-on-Demand. Użytkownik ma do dyspozycji wszystkie funkcje terminala 5250 znane z produktu IBM i Access for Windows oraz pewne funkcje dodatkowe:
    • Obsługa kart, która pozwala na współużytkowanie pojedynczego obszaru wyświetlania przez wiele sesji
    • Obsługa kółka myszy, która pozwala na intuicyjne przewijanie stron
    • Przełączanie między językami nie wymaga restartowania stacji roboczej
    • Współbieżne sesje terminala o różnych stronach kodowych hosta, które pozwalają na użycie osobnych języków w różnych sesjach emulatora
  • Emulacja drukarki
  • Menedżer sesji 5250 utworzony na podstawie menedżera sesji IBM Personal Communications, którego można używać do zarządzania wszystkimi sesjami emulatora 5250.
  • Funkcja przesyłania danych, która udostępnia możliwość przesyłania danych z/do bazy danych systemu IBM i, z/do różnych typów plików na stacji roboczej, takich jak arkusz kalkulacyjny OpenDocument (*.ods), skoroszyt programu Excel (*.xlsx) i inne.
  • Obszar Wydruk udostępnia interfejs umożliwiający wyświetlanie plików znajdujących się w kolejkach wyjściowych systemu IBM i oraz pobieranie tych plików do systemu klienta.
  • Wirtualny panel sterowania z interfejsem graficznym panelu operacyjnego systemu IBM i.
  • Emulacja terminala 5250 konsoli LAN, HMC i FSM.
  • Konsolidacja konfiguracji interfejsu zarządzania sprzętem, w tym ASMI, IVM i HMC.
  • Możliwość uruchomienia produktu IBM Navigator for i za pomocą domyślnej przeglądarki.
  • Funkcje bazy danych, takie jak Uruchamianie skryptów SQL i Centrum wydajności SQL.

Opcjonalny pakiet aplikacji systemu Windows udostępnia następujące funkcje:

  • Połączenie z bazą danych Db2 for i za pomocą sterownika ODBC oraz dostawców .NET i OLE DB
  • Pakiet programistyczny służący do uzyskiwania dostępu do obiektów systemu IBM i
  • Obsługa połączeń TLS i SSL
  • Sterownik drukarki AFP

Opcjonalny pakiet aplikacji systemu Linux udostępnia następujące funkcje:

  • Sterownik ODBC służący do uzyskiwania dostępu do bazy danych Db2 for i oraz obsługa pełnych 64-bitowych typów danych sterownika ODBC.

3.1 Wymagania wstępne (stacja robocza)

Produkt IBM i Access Client Solutions działa w większości systemów operacyjnych obsługujących środowisko Java 8.0 lub jego nowszą wersję, w tym w różnych wersjach systemów Linux, Mac i Windows.

Zalecenie:
Korzystanie zawsze z najnowszej wersji środowiska Java zagwarantuje posiadanie zawsze wszystkich najnowszych poprawek i poprawek zabezpieczeń.

Jeden ze sposobów sprawdzenia wersji środowiska Java zainstalowanego w systemie polega na wywołaniu zachęty do wprowadzenia komendy (wiersz komend, powłoka, terminal itd.), a następnie wpisaniu komendy:
    java -version

Następujące dane wyjściowe wskazują, że została zainstalowana wersja 8.0:
    java version "1.8.0_65"
Liczba 65 w tym przykładzie odnosi się do poziomu aktualizacji.

Poniżej wymieniono niektóre serwisy WWW dostawców środowiska Java. Należy sprawdzić, czy używana jest najnowsza wersja środowiska Java przeznaczona dla danej platformy.

5.0 Instalacja

Pakiet produktu IBM i Access Client Solutions udostępnia kilka opcji instalacji, od bardzo prostej instalacji dla jednego użytkownika do bardziej zaawansowanej instalacji dla wielu użytkowników.

Skrypty instalacyjne znajdują się w folderach Mac_Application, Linux_Application i Windows_Application (patrz sekcja 4.0 Zawartość produktu). Można ich używać na potrzeby wielu różnych opcji instalacji.

W przypadku instalacji pojedynczego użytkownika w systemach Mac, Linux i Windows lub instalacji dla wielu użytkowników na komputerze PC z systemem Windows należy zapoznać się z publikacją Szybki start. Publikacja Szybki start wyjaśnia, w jaki sposób można użyć udostępnionych skryptów instalacyjnych do wykonywania tych typów prostej instalacji.

Administrator planujący wdrożenie dla wielu użytkowników powinien zapoznać się z artykułem IBM i Access Client Solutions: Customization and deployment made easy (IBM i Access Client Solutions: Łatwa konfiguracja i wdrożenie). W artykule opisano, w jaki sposób administrator może wstępnie dostosować instalację dla wielu użytkowników przed wdrożeniem za pomocą opcji /AdminConfig obsługiwanej przez skrypty instalacyjne.

Administratorzy, którzy chcą zautomatyzować instalację w trybie cichym, mogą użyć parametru /AdminConfig, aby wstępnie skonfigurować instalację. Następnie należy użyć parametru /Q podczas rzeczywistej instalacji. Na przykład:

    Windows_Application\install_acs_64.js /AdminConfig
    Windows_Application\install_acs_64.js /Q

Oprócz powyższych opcji istnieje także możliwość rozpakowywania archiwum zip w dowolnym wybranym położeniu. Może to być dowolne miejsce, do którego stacja robocza ma uprawnienia do odczytu plików. W szczególności może to być lokalny dysk twardy, zdalny dysk sieciowy (współużytkowany) lub nośnik przenośny, taki jak dysk CD lub flash USB. Rozpakowanie archiwum zip powoduje zakończenie instalacji.

Uwaga techniczna:
Niektóre programy narzędziowe do archiwizowania nie zachowują wszystkich atrybutów zapisanych plików. Na przykład na platformach Mac i Linux użycie komendy unzip jest z reguły lepszym rozwiązaniem niż użycie komendy jar. Więcej informacji na ten temat zawiera sekcja 6.0 Uprawnienia do plików.

5.1 Aktualizacja istniejącej instalacji

Okresowo będą udostępniane rozszerzenia i poprawki. Te aktualizacje są dostarczane w postaci pełnej instalacji produktu. Po udostępnieniu aktualizacji należy zaktualizować istniejącą instalację.

W celu zaktualizowania istniejącej instalacji, która została wykonana przy użyciu skryptów instalacyjnych, należy wyodrębnić zawartość nowszej wersji produktu. Skrypt instalacyjny należy wywołać z poziomu tej nowej wersji w taki sam sposób, jak miało to miejsce podczas początkowej instalacji. Spowoduje to zaktualizowanie plików produktu bez wpływu na istniejącą konfigurację użytkowników.

Administratorzy obsługujący wersję produktu w położeniu centralnym, do którego ma dostęp wielu użytkowników, mogą wyodrębnić zawartość nowszej wersji, nadpisując wersję istniejącą. Jeśli wprowadzono niestandardowe zmiany w pliku AcsConfig.properties, należy zapisać plik AcsConfig.properties przed wyodrębnieniem zawartości nowej wersji produktu na istniejącej. Po wyodrębnieniu zawartości nowej wersji produktu i przywróceniu pliku AcsConfig.properties użytkownicy, którzy chcą zaktualizować swoją instalację, mogą uruchomić skrypt instalacyjny w taki sam sposób, jak podczas początkowej instalacji.

Jeśli wybrano opcję zainstalowania produktu bez użycia skryptów instalacyjnych przez wyodrębnienie zawartości archiwum zip w wybranym położeniu, należy w celu zaktualizowania produktu wyodrębnić zawartość nowszej wersji produktu na wersji istniejącej. Należy pamiętać, aby zapisać i przenieść zawartość pliku AcsConfig.properties, jeśli wprowadzono do niego niestandardowe zmiany.

6.0 Uprawnienia do plików

W sekcji 7.0 Uruchamianie produktu opisano kilka różnych sposobów uruchamiania produktu IBM i Access Client Solutions. W przypadku uruchamiania produktu za pomocą jednego z udostępnionych plików binarnych lub skryptów należy sprawdzić, czy dla danego pliku zostały włączone uprawnienia do wykonania. Uprawnienia do pliku przypisywane podczas rozpakowywania pliku archiwum zip zależą od kilku czynników, w tym od systemu operacyjnego, programu narzędziowego używanego do rozpakowywania archiwum zip, uprawnień użytkownika itp.

W przypadku problemów z korzystaniem z jednego z udostępnionych plików binarnych lub skryptów należy sprawdzić uprawnienia do plików. W następujących sekcjach opisano niektóre metody sprawdzania uprawnień do plików.

6.1 Uprawnienia do plików (Linux, Mac, AIX)

W systemach uniksowych można użyć następującej komendy z poziomu zachęty powłoki lub terminala w celu sprawdzenia uprawnień do pliku:

    ls -l <plik>

Aby zmienić uprawnienia do pliku, można użyć następującej komendy:

    chmod <uprawnienie> <plik>

Na przykład:
aby dodać uprawnienie do odczytu i wykonania dla wszystkich użytkowników: 
    chmod a+rx <plik>
aby nadać właścicielowi pliku uprawnienia do odczytu, zapisu i wykonywania, a dla wszystkich innych użytkowników tylko uprawnienia do odczytu i wykonywania: 
    chmod 755 <plik>

Dodatkowa pomoc dla komend ls i chmod jest łatwo dostępna w Internecie.

6.2 Uprawnienia do plików (Windows)

W systemie Windows podczas przeglądania plików za pomocą Eksploratora Windows należy kliknąć plik prawym przyciskiem myszy i wybrać opcję Właściwości. Karta Zabezpieczenia powinna zawierać uprawnienia do plików. Użytkownik powinien mieć uprawnienia do odczytu i wykonywania.

W najnowszych wersjach systemu Windows do wyświetlania i zmiany uprawnień do plików można użyć również komendy icacls.

7.0 Uruchamianie produktu

Istnieje wiele sposobów uruchamiania produktu IBM i Access Client Solutions. Jeśli użyto skryptów instalacyjnych do instalowania produktu, najłatwiejszy sposób uruchomienia produktu można znaleźć w publikacji Szybki start.

Jeśli produkt nie był instalowany przy użyciu skryptów instalacyjnych, w dalszej części tej sekcji opisano alternatywne sposoby uruchamiania produktu. Ze względu na to, że środowisko Java może być zainstalowane na wiele sposobów i w wielu położeniach, niektóre z tych metod mogą wymagać dodatkowej konfiguracji. Jeśli jedna z poniższych metod nie działa, należy użyć innej. W niektórych przypadkach zostały udostępnione dodatkowe wskazówki. Pomocna może być również sekcja 11.0 Często zadawane pytania.

Jeśli używany jest plik binarny lub skrypt (zgodnie z poniższym opisem), musi on znajdować się w strukturze katalogów zawartej w archiwum zip. Dla wygody można również skopiować lub przenieść pliki binarne i/lub skrypty przeznaczone dla danych platform do tego samego katalogu, w którym znajduje się plik acsbundle.jar.

7.1 Uruchamianie produktu

Aby uruchomić produkt z poziomu przeglądarki plików (np. Eksploratora Windows, programu Finder w systemie Mac OS X itp.) za pomocą pliku binarnego specyficznego dla platformy, w katalogu Start_Programs należy znaleźć podkatalog, który identyfikuje system operacyjny i architekturę sprzętową.

Użytkownik powinien znaleźć plik binarny, który jest rozpoznawany przez system operacyjny. Następnie należy kliknąć dwukrotnie ten plik, aby uruchomić produkt. Ponadto można uruchomić produkt przy użyciu tego pliku binarnego z poziomu wiersza komend, terminala lub powłoki.

Jeśli wystąpi następujący błąd:
    „Błąd ładowania modułu Java.”
produkt IBM i Access Client Solutions nie może znaleźć instalacji Java w rozpoznanym położeniu. Należy wypróbować jedną z następujących metod opisanych w sekcjach:
    7.1.1 Uruchamianie produktu – opcje dodatkowe
    7.2 Uruchamianie produktu (za pomocą skryptu)

7.2 Uruchamianie produktu (za pomocą skryptu)

Począwszy od wydania v1r1m5, nazwa katalogu Start_Scripts została zmieniona na Sample_Scripts, a sam katalog został umieszczony w katalogu Documentation.

Preferowanym sposobem uruchamiania produktu jest użycie specyficznego dla platformy pliku binarnego dostępnego w folderze Start_Programs. Skryptów w folderze Sample_Scripts należy używać tylko wtedy, gdy nie działa plik binarny przeznaczony dla danej platformy.

Aby uruchomić produkt z poziomu przeglądarki systemu plików (np. Eksploratora Windows, programu Finder systemu Mac OS X itp.) przy użyciu jednego z dostarczonych skryptów, w podkatalogu Sample_Scripts należy znaleźć skrypt, który jest zgodny z systemem operacyjnym użytkownika.

W większości systemów operacyjnych opartych na platformie innej niż Windows jest domyślnie dostępny język Perl. Katalog Sample_Scripts/Linux_Mac_Other zawiera skrypt w języku Perl (z trzema różnymi rozszerzeniami nazwy pliku), którego można używać do uruchamiania produktu na dowolnej platformie, na której język Perl jest dostępny. Należy wybrać plik z rozszerzeniem rozpoznawalnym przez system operacyjny jako skrypt języka Perl.

W systemach opartych na platformie Windows domyślnie jest dostępny język JScript. Katalog Sample_Script/Windows zawiera plik JScript, którego można używać do uruchamiania produktu w systemach operacyjnych Windows.

Przeglądając system plików przy użyciu metody specyficznej dla platformy (np. Eksploratora Windows, programu Finder systemu Mac OS X itp.), należy znaleźć skrypt rozpoznawalny przez system operacyjny. Następnie należy kliknąć go dwukrotnie, aby uruchomić produkt. Ponadto można uruchomić produkt przy użyciu tego skryptu z poziomu wiersza komend, terminala lub powłoki.

7.3 Uruchamianie produktu (za pomocą wiersza komend)

Produkt można również uruchomić z poziomu wiersza komend z dowolnego miejsca, w którym można wprowadzić komendę (wiersz komend, terminal, powłoka itp.)

    java -Xmx1024m -jar <ścieżka>/acsbundle.jar
gdzie <ścieżka> jest położeniem wykonywalnego pliku jar produktu

Na przykład:
    java -Xmx1024m -jar V:/jakieś_położenie/acsbundle.jar
lub 
    <ścieżka_do_środowiska_Java>java -jar V:/jakieś_położenie/acsbundle.jar
gdzie <ścieżka_do_środowiska_Java> to położenie komendy java dla pakietu JDK 6.0 lub nowszego.

Aby określić pełną ścieżkę do komendy java, należy zapoznać się z sekcją 7.1.2 Znajdowanie ścieżki do katalogu głównego środowiska Java.

Możliwe jest również użycie dowolnego z programów lub skryptów z poziomu wiersza komend Na przykład:

    /Położenie/produktu/Start_Programs/Linux_x86-64/acslaunch_linux-64
    /Położenie/produktu/Start_Programs/Mac_i386-32_x86-64/acslaunch_mac
    C:\Położenie\produktu\Start_Programs\Windows_x86-64\acslaunch_win-64.exe

Uwaga techniczna:
Na większości platform domyślna maksymalna wielkość obszaru sterty wirtualnej maszyny języka Java jest zbyt mała, aby można było używać wielu funkcji produktu IBM i Access Client Solutions. Zalecaną minimalną wielkością jest sterta o maksymalnej wielkości jednego gigabajta (-Xmx1024m). Określenie wielkości mniejszych niż jeden gigabajt lub użycie domyślnej wielkości sterty może spowodować wystąpienie wyjątku OutOfMemoryException.

8.0 Konfiguracja

Konfigurację systemu należy dodać dla każdego systemu IBM i, który będzie używany lub zarządzany przez użytkownika. Aby dodać konfigurację systemu, należy wybrać zadanie Konfiguracje systemu dostępne w ramach węzła zadań Zarządzanie. Następnie należy wybrać opcję Nowa. Na karcie Ogólne należy wprowadzić nazwę systemu. Aby rozpocząć wykonywanie zadań ogólnych, wystarczy tylko wprowadzić nazwę systemu.

Po zakończeniu należy kliknąć przycisk OK, aby zapisać informacje wprowadzone dla tego systemu, lub wybrać opcję Zapisz/Nowy, jeśli istnieją dodatkowe systemy, które mają zostać dodane do konfiguracji.

Istnieje możliwość dodania nowych systemów do konfiguracji lub zaktualizowania istniejących konfiguracji w dowolnym czasie za pomocą kart Ogólne, Połączenie lub Konsola.

W przypadku zadań konsoli wymagana jest dodatkowa konfiguracja. Konfiguracje konsoli są automatycznie wiązane z nazwą systemu wprowadzoną na karcie Ogólne. Aby wprowadzić konfigurację konsoli dla systemu, należy wybrać zadanie Konfiguracje systemu dostępne w ramach węzła zadań Zarządzanie. Następnie należy wybrać opcję Nowa lub Edytuj. Następnie należy wybrać kartę Konsola. Zadanie Konsola 5250 wymaga skonfigurowanej konsoli sieci LAN lub konsoli HMC. Jeśli konsola LAN lub HMC nie jest skonfigurowana, należy zapoznać się z informacjami w sekcji 9.9 Nawiązywanie połączenia konsoli z systemem IBM i.

Zadanie Interfejs zarządzania sprzętem wymaga skonfigurowanego interfejsu zarządzania sprzętem. Użytkownik może wprowadzić maksymalnie dwie konfiguracje interfejsu zarządzania sprzętem.

Po zakończeniu należy wybrać opcję Zamknij na panelu Konfiguracje systemu.

Za pomocą pola rozwijanego System znajdującego się na głównym panelu produktu IBM i Access Client Solutions należy wybrać system. Wszystkie zadania konsoli automatycznie powiążą wybrany system wprowadzony na karcie Ogólne z konfigurację konsoli wprowadzoną na karcie Konsola.

Teraz można wybrać zadanie dla wybranego systemu. Jeśli zostanie wybrane zadanie konsoli, dla którego nie wprowadzono odpowiednich informacji na karcie konfiguracji Konsola, zostanie wyświetlony komunikat o błędzie.

9.2 Powiązania plików

Niektóre pliki konfiguracyjne wygenerowane przez produkt IBM i Access Client Solutions są obsługiwane w wierszu komend wtedy, gdy zostaną podane jako pierwszy i jedyny parametr. Jeśli te pliki o określonych rozszerzeniach nazwy zostaną podane jako pierwszy parametr, produkt IBM i Access Client Solutions powiąże plik z wywoływaną funkcją i udostępni go jako dane wejściowe tej funkcji.

Obsługa powiązań plików jest dostępna dla następujących rozszerzeń: 

    .dttx   – żądanie przesyłanie w funkcji przesyłania danych
    .dtfx   - żądanie pobierania w funkcji przesyłania danych
    .hod    - profil sesji emulatora 5250
    .bchx   - profil emulacji wielu sesji
    .ws     - profil sesji emulatora 5250 (Personal Communications)
    .bch    - profil emulacji wielu sesji (Personal Communications)
Przykłady wiersza komend: 
    acslaunch_xxx dt_download_file.dtfx – uruchamia zapisaną operację pobierania
    acslaunch_xxx dt_upload_file.dttx   – uruchamia zapisaną operację przesyłania
    acslaunch_xxx system_lp13ut20.hod   – uruchamia sesję 5250 z systemem
gdzie acslaunch_xxx to składnia wiersza komend używana do uruchamiania produktu. Patrz sekcja 7.3 Uruchamianie produktu (za pomocą wiersza komend).

Te obsługiwane przez wiersz komend powiązania plików umożliwiają użytkownikowi ręczne skonfigurowanie powiązań plików specyficznych dla systemu operacyjnego. Ponieważ powiązania plików są zależne od platformy, wymagane kroki zależą od systemu operacyjnego.

Skonfigurowanie powiązań plików dla danego systemu operacyjnego umożliwia uruchomienie wyznaczonej funkcji przez dwukrotne kliknięcie pliku jednego z powyższych obsługiwanych typów.

W następujących sekcjach przedstawiono przykłady konfigurowania powiązań plików dla niektórych systemów operacyjnych.

9.2.1 Powiązania plików (w systemie Windows)

  1. Na pasku menu głównego interfejsu GUI wybierz opcję Narzędzia -> Powiązania plików...
  2. Wybierz typy plików, dla których chcesz utworzyć powiązania.
  3. Kliknij przycisk OK.

Po dwukrotnym kliknięciu plików tego typu będzie teraz uruchamiana odpowiednia funkcja produktu IBM i Access Client Solution.

9.2.1.1 Zmiana ikony (dla skrótu w systemie Windows)

  1. Znajdź skrót utworzony wcześniej dla pliku binarnego odpowiedniego w danej architekturze sprzętowej (acslaunch_win-32.exe LUB acslaunch_win-64.exe).
  2. Kliknij go prawym przyciskiem myszy i wybierz opcję Właściwości.
  3. Wybierz kartę Skrót.
  4. Kliknij przycisk Zmień ikonę...
  5. Wybierz ikonę, która ma być używana dla skrótu.
  6. Kliknij przycisk OK.
  7. Kliknij przycisk OK.

9.2.2 Konfigurowanie ikony pulpitu (w systemie Linux)

Należy wykonać kroki przedstawione w publikacji Szybki start dla systemu Linux. Dzięki temu aplikacja zostanie zainstalowana w katalogu /opt/ibm/iAccessClientSolutions. Utworzony zostanie również plik /usr/share/applications/IBM i Accesss Client Solutions.desktop.

Aby utworzyć ikonę produktu na pulpicie, należy skopiować powyższy plik .desktop do folderu pulpitu. Może być konieczne dostosowanie uprawnień do pliku na pulpicie, aby był on wykonywalny.

9.2.2.1 Powiązania plików (w systemie Linux)

Kroki wymagane do skonfigurowania powiązań plików zależą od dystrybucji systemu Linux i używanego środowiska pulpitu. W zasadzie wymagane kroki są podobne do powyższych kroków używanych w przypadku systemu Windows.

9.3 Zmiana położenia konfiguracji

Domyślnie położenie konfiguracji dla każdego użytkownika jest unikalne. Położenie konfiguracji można zmienić, ustawiając właściwość:
    com.ibm.iaccess.AcsBaseDirectory
Ta właściwość znajduje się w pliku AcsConfig.properties.

Plik AcsConfig.properties w dostarczanym produkcie można znaleźć w dwóch miejscach. Jest on zawarty w pliku acsbundle.jar. Dla wygody jest on również udostępniany w pliku archiwum zip produktu i po jego rozpakowaniu będzie znajdować się w tym samym katalogu co plik acsbundle.jar.

Podczas uruchamiania produkt będzie korzystał tylko z pierwszego znalezionego pliku AcsConfig.properties. Najpierw zostanie sprawdzony katalog zawierający plik acsbundle.jar. Jeśli plik AcsConfig.properties nie zostanie znaleziony w tym samym katalogu co plik acsbundle.jar, produkt użyje pliku AcsConfig.properties z pliku acsbundle.jar.

Plik AcsConfig.properties w archiwum acsbundle.jar może zostać zaktualizowany przy użyciu ścieżki konfiguracji niestandardowej. W tym przypadku należy upewnić się, że katalog z plikiem acsbundle.jar nie zawiera pliku AcsConfig.properties. W przeciwnym razie zostanie użyty właśnie ten plik. Dzięki temu można w elastyczny sposób decydować, czy położenie konfiguracji ma być dystrybuowane z plikiem acsbundle.jar, czy przesłaniane.

Podczas definiowania ścieżki konfiguracji można korzystać z udostępnionych specjalnych słów kluczowych. Gdy te słowa kluczowe będą znajdować się w określonej ścieżce, zostaną zastąpione definiowanymi przez nie tekstami lub ścieżkami. Ścieżka konfiguracji może zawierać tylko jedno słowo kluczowe. Poniżej podano specjalne słowa kluczowe oraz opisy ich znaczeń:

	{USER} – identyfikator bieżącego użytkownika. To słowo kluczowe można stosować
w dowolnym miejscu ścieżki.

Następujące słowa kluczowe mogą znajdować się tylko na początku określonej ścieżki: 

	{PRODUCTDIR} – ścieżka do katalogu z plikiem acsbundle.jar
	{TEMPDIR}    – ścieżka do katalogu tymczasowego specyficznego dla platformy
	{ROOT}       – ścieżka do katalogu głównego systemu plików
	{HOME}       – ścieżka do katalogu osobistego użytkownika
	{DEFAULT}    – ścieżka domyślna, która byłaby zwykle używana przez produkt

Uwaga techniczna:
Nie jest zalecane współużytkowanie konfiguracji przez wielu użytkowników. Jeśli na przykład X byłby współużytkowanym dyskiem sieciowym, następujące ustawienie może spowodować nieprzewidywalne rezultaty: 

    com.ibm.iaccess.AcsBaseDirectory=X:/współużytkowany_dysk_sieciowy/katalog_konfiguracji

Współużytkowanie ścieżki konfiguracji przez wielu użytkowników może spowodować następujące problemy:

  1. Jednoczesne używanie konfiguracji przez wielu użytkowników może przynieść nieprzewidywalne rezultaty.
  2. W tym przykładzie przyjęto, że każdy użytkownik ma napęd X przypisany do tego samego położenia.
  3. W przypadku klientów w systemach Linux lub Mac nie działa konfiguracja z dyskiem sieciowym. W systemach Windows, Linux i Mac będą działać słowa kluczowe (jak w poniższych przykładach).

W przypadku współużytkowania ścieżki konfiguracji przez wielu użytkowników należy użyć słowa kluczowego {USER} w celu uniknięcia kolizji z innymi użytkownikami. Słowo kluczowe zostanie zastąpione identyfikatorem bieżącego użytkownika.

Podczas ustawiania ścieżki konfiguracji jako separatora katalogów należy użyć ukośnika ('/') zamiast ukośnika odwrotnego ('\'). Takie rozwiązanie będzie działać we wszystkich systemach operacyjnych, łącznie z systemem Windows.

Przykładowe zalecane konfiguracje:

Przykład 1 – konfiguracja lokalna dla bieżącego użytkownika (domyślnie): 

    com.ibm.iaccess.AcsBaseDirectory=
Jeśli nie ustawiono właściwości AcsBaseDirectory, wartością domyślną konfiguracji dla użytkownika jest ścieżka zależna od platformy. Jest to ustawienie domyślne produktu IBM i Access Client Solutions.

Przykład 2 – konfiguracja zdalna (lub lokalna) unikalna dla każdego użytkownika: 

    com.ibm.iaccess.AcsBaseDirectory={ROOT}/katalog_konfiguracyjny/{USER}/
Konfiguracja będzie zdalna lub lokalna – zgodnie z położeniem katalogu {ROOT}.

Przykład 3 – konfiguracja zdalna (lub lokalna) unikalna dla każdego użytkownika: 

    com.ibm.iaccess.AcsBaseDirectory={PRODUCTDIR}/katalog_konfiguracyjny/{USER}/
Konfiguracja będzie zdalna lub lokalna – zgodnie z położeniem katalogu {PRODUCTDIR}.

Przykład 4 – konfiguracja lokalna na nośniku przenośnym (takim jak dysk USB): 

    com.ibm.iaccess.AcsBaseDirectory={PRODUCTDIR}/katalog_konfiguracyjny
Ponieważ ścieżka do nośnika przenośnego różni się w zależności od systemu, w którym jest on używany, to ustawienie pozwala na określenie konfiguracji względem plików produktu. W tym przykładzie celowo nie podano słowa kluczowego {USER}, więc z konfiguracji na dysku USB będzie korzystać jego użytkownik niezależnie od identyfikatora użytkownika.

9.4.1 Automatyczny import ustawień konfiguracji

Poniższe właściwości mogą być używane w pliku AcsConfig.properties do automatycznego określania konfiguracji dla nowych użytkowników lub aktualizowania konfiguracji dla istniejących użytkowników:

   com.ibm.iaccess.autoimport
   com.ibm.iaccess.autoimport.version

Poniżej podano wymagane kroki:

  1. Utwórz konfigurację, która ma zostać propagowana dla jednego lub większej liczby użytkowników.
  2. Wyeksportuj konfigurację do pliku, używając opcji Plik -> Eksport konfiguracji znajdującej się w głównym interfejsie GUI, lub użyj opcji wiersza komend opisanej w sekcji 9.1.1 Backup.
  3. Przenieś plik konfiguracyjny w pożądane miejsce. Dostępnych jest kilka opcji, które zostaną opisane później.
  4. Ustaw właściwość com.ibm.iaccess.autoimport w pliku AcsConfig.properties na ścieżkę do pliku konfiguracyjnego.
  5. Ustaw właściwość com.ibm.iaccess.autoimport.version na wartość całkowitą reprezentującą wersję pliku konfiguracyjnego.

Sposób działania jest następujący:
Zapisana konfiguracja przywoływana przez właściwość com.ibm.iaccess.autoimport zostanie automatycznie zaimportowana, gdy wartość właściwości com.ibm.iaccess.autoimport.version nie będzie zgodna z wartością ostatniej zaimportowanej wersji. Oprócz możliwości ustawienia początkowej konfiguracji i aktualizacji istniejącej konfiguracji, pozwala to również na przywrócenie wcześniejszych wersji konfiguracji. Konfiguracja zostanie zaktualizowana za każdym razem, gdy wystąpi niezgodność między wartością całkowitą właściwości wersji a wartością ostatniej zaimportowanej wersji. Jednakże, mimo że zaimportowana konfiguracja może spowodować zmiany konfiguracji dla istniejącego systemu w konfiguracji użytkownika, system nigdy nie zostanie usunięty z konfiguracji użytkownika.

Ścieżka do właściwości com.ibm.iaccess.autoimport może zostać określona jako ścieżka bezwzględna, adres URL lub ścieżka ze słowami kluczowymi zdefiniowanymi w sekcji 9.3. Na przykład: 

    com.ibm.iaccess.autoimport=C:/acs_bak.zip

    com.ibm.iaccess.autoimport=file///C:/acs_bak.zip

    com.ibm.iaccess.autoimport=http://firma.com/ścieżka/plik/acs_bak.zip

    com.ibm.iaccess.autoimport=ftp://firma.com/ścieżka/plik/acs_bak.zip

    com.ibm.iaccess.autoimport={PRODUCTDIR}/acs_bak.zip

Możliwość dystrybucji pliku konfiguracyjnego w pliku acsbundle.jar lub w katalogu zawierającym plik acsbundle.jar zapewnia dodatkową elastyczność. W obu przypadkach należy ustawić właściwość com.ibm.iaccess.autoimport na nazwę pliku bez poprzedzającej ścieżki:

    com.ibm.iaccess.autoimport=acs_bak.zip

Dozwolona jest wartość specjalna (*) właściwości com.ibm.iaccess.autoimport.version: 

    com.ibm.iaccess.autoimport.version=*

Spowoduje to, że konfiguracja będzie zawsze importowana, niezależnie od numeru poprzedniej wersji.

9.4.2 Własne wdrożenia systemu IBM i

Niektóre wtyczki wiersza komend, które nie wymagają interfejsu GUI, mogą być używane w trybie własnym w systemie IBM i. Na przykład można użyć funkcji przesyłania danych do wyodrębniania danych z bazy danych do pliku arkusza kalkulacyjnego bezpośrednio w systemie IBM i bez pobierania danych na komputer.

W tym celu można wyodrębnić zawartość archiwum zip produktu w dowolnym miejscu zintegrowanego systemu plików (Integrated File System - IFS) systemu IBM i. Jeśli na przykład wyodrębniono archiwum zip produktu w katalogu: 

 /home/AccessClientSolutions
A wcześniej zapisano żądanie pobierania funkcji przesyłania danych, które istnieje w katalogu: 
 /ścieżka/qcustcdt.dtfx
Można uruchomić następującą komendę w systemie IBM i: 
QSH CMD('java -jar /home/AccessClientSolutions/acsbundle.jar /plugin=dtbatch /ścieżka/qcustcdt.dtfx')

9.6 Migracja z produktu IBM i Access for Windows

Pliki konfiguracyjne produktu IBM i Access Client Solutions nie są kompatybilne z odpowiadającymi im funkcjami produktu IBM i Access for Windows. Produkt IBM i Access Client Solutions udostępnia ścieżkę migracji dla kilku ważnych elementów, co zostało przedstawione w następujących sekcjach.

9.6.1 Migracja konfiguracji systemu

Funkcja kopiowania połączeń dostępna w menu głównego interfejsu GUI (po wybraniu opcji Plik -> Kopiuj połączenia) udostępnia interfejs do kopiowania konfiguracji systemu między produktem IBM i Access Client Solutions i wcześniejszymi konfiguracjami systemu Windows obsługiwanymi przez produkt IBM i Access for Windows. Informacje dodatkowe można znaleźć w pomocy do panelu głównego funkcji Kopiowanie połączeń. Konfiguracje systemu można też migrować przy użyciu wiersza komend. Informacje dodatkowe zawiera sekcja 9.1.25 MIGRATE.

9.6.2 Migracja emulacji terminala 5250

Pliki emulacji terminala 5250 używane przez emulator Personal Communications produktu IBM i Access for Windows można przekształcić przy użyciu menedżera sesji 5250 w produkcie IBM i Access Client Solutions. Następujące typy plików można przekształcić z formatu produktu Personal Communications:

   .ws   – profil emulatora
   .bch  – plik wsadowy emulatora
   .kmp  – plik dostosowania klawiatury
   .pmp  – pliki podręcznej planszy klawiszy
   .bar  - pliki paska menu
Pliki zostaną przekształcone w następujący format:
   .hod  – profil emulatora
   .bchx – plik wsadowy emulatora
   .kmp  – plik dostosowania klawiatury
   .pmp  – pliki podręcznej planszy klawiszy
   .bar  - pliki paska menu
Typy plików .kmp, .pmp i .bar są używane przez obydwa produkty. Formaty nie są jednak kompatybilne. Przekonwertowane pliki zostaną utworzone w ścieżce specyficznej dla produktu IBM i Access Client Solutions.

Konwersję tych plików można zainicjować z menu menedżera sesji programu IBM i Access Client Solutions:

  • wybierając opcję Plik->Import, a następnie znajdując pliki produktu Personal Communications;
  • przeciągając plik i upuszczając go w oknie menedżera sesji produktu IBM i Access Client Solutions
...lub z wiersza komend za pomocą wtyczki PM5250. Więcej informacji zawiera sekcja 9.1.30 PM5250.

Dostępny jest program narzędziowy do przekształcania makr. Z poziomu menedżera sesji należy wybrać opcję:

    Narzędzia->Przekształć makro...
Uwaga techniczna:
Program narzędziowy do przekształcania makr może nie być w stanie przekształcić makra PC5250, jeśli jest to makro niestandardowe zawierające więcej niż tylko zarejestrowane naciśnięcia klawiszy.

9.6.3 Migracja zapisanych plików żądań przesyłania danych

Funkcja przesyłania danych produktu IBM i Access Client Solutions udostępnia kreator służący do przekształcania zapisanych plików żądań przesyłania danych, które zostały wygenerowane przez produkt IBM i Access for Windows.

Z produktu IBM i Access for Windows można przekształcić następujące typy plików:

   .dtf – przesyłanie danych z systemu IBM i
   .dtt – przesyłanie danych do systemu IBM i
Pliki zostaną przekształcone w następujący format: 
   .dtfx – przesyłanie danych z systemu IBM i
   .dttx – przesyłanie danych do systemu IBM i
Kreator migracji funkcji przesyłania danych nie umożliwia migracji plików .fdf. Nowy typ pliku wymagany przez produkt IBM i Access Client Solutions to .fdfx. Można go wygenerować podczas pobierania lub przy użyciu kreatora tworzenia plików.

Kreator migracji jest dostępny w menu głównym funkcji przesyłania danych po wybraniu opcji Działania -> Migracja przesyłania danych.

9.8 Przesyłanie danych

9.8.1 Obsługa funkcji przesyłania danych dla arkuszy kalkulacyjnych programu Excel i Calc

Funkcja przesyłania danych, oprócz możliwości pobierania danych do pliku, obsługuje także pobieranie do aktywnego arkusza kalkulacyjnego programu Microsoft Excel lub OpenOffice Calc. W celu pobrania danych do aktywnego arkusza kalkulacyjnego na panelu interfejsu GUI funkcji przesyłania danych udostępniono opcję wyboru urządzenia wyjściowego. Domyślnie urządzeniem wyjściowym jest plik. Jeśli używana platforma obsługuje interakcję z arkuszami kalkulacyjnymi programu Excel i/lub Calc, dodatkowe opcje dla aktywnego arkusza kalkulacyjnego programu Excel i Calc mogą zostać wybrane z pola rozwijanego Urządzenie wyjściowe.

Ograniczenia:

  • W systemie Windows wymagana jest wersja przynajmniej OpenOffice 3.4.1.
  • W systemach Linux i Mac wymagana jest wersja przynajmniej OpenOffice 4.2.1 i oprogramowanie to musi być zainstalowane w położeniu domyślnym.
  • Produkt IBM i Access Client Solutions działający w systemie Linux lub Mac obsługuje wyłącznie interakcje z oprogramowaniem OpenOffice.
    Produkt IBM i Access Client Solutions działający w systemie WIndows obsługuje interakcje z programami Microsoft Excel i OpenOffice Calc.
  • Architektura bitowa aplikacji OpenOffice musi być zgodna z architekturą bitową produktu IBM i Access Client Solutions.
    Objaśnienie: oprogramowanie OpenOffice jest dostępne w wersjach 32- i 64-bitowych. Obie wersje mogą działać na 64-bitowym komputerze PC. Podczas pracy z arkuszem kalkulacyjnym OpenOffice architektura bitowa aplikacji OpenOffice i produktu IBM i Access Client Solutions muszą być takie same. W przeciwnym razie interakcja funkcji przesyłania danych z arkuszem kalkulacyjnym nie będzie możliwa. Architektura bitowa aplikacji IBM i Access Client Solutions jest determinowana przez wirtualną maszynę języka Java (JVM), na której ta aplikacja jest uruchomiona. W przypadku wystąpienia niezgodności wersji między aplikacją OpenOffice a maszyną JVM, na której jest uruchomiony produkt IBM i Access Client Solutions, i wybraniu aktywnego arkusza kalkulacyjnego jako urządzenia wyjściowego, zostanie wyświetlony błąd podobny do błędu Nie znaleziono aktywnego arkusza kalkulacyjnego. Jeśli praca odbywa się na 64-bitowym komputerze PC, produkt IBM i Access Client Solutions można uruchomić za pomocą 32-bitowego lub 64-bitowego programu uruchamiającego pliki binarne specyficznego dla używanej platformy. Aby obsługa aktywnego arkusza kalkulacyjnego działała, należy uruchomić produkt IBM i Access Client Solutions za pomocą pliku binarnego specyficznego dla platformy, który jest zgodny z architekturą bitową używanej aplikacji OpenOffice.
    Ponieważ system Mac obsługuje jedynie 64-bitową maszynę JVM, w systemie Mac należy zainstalować 64-bitową wersję programu OpenOffice w celu korzystania z aktywnej obsługi arkuszy kalkulacyjnych.

9.8.2 Obsługa obcinania znaków i nadmiaru liczbowego w funkcji przesyłania danych

Podczas żądania przesyłania w funkcji przesyłania danych, jeśli pole znakowe lub liczbowe przekracza zdefiniowaną wielkość pola, żądanie przesyłania zostaje zakończone.

Aby włączyć obcinanie pól znakowych na końcu, należy ustawić następującą właściwość w pliku AcsConfig.properties:
    com.ibm.iaccess.dataxfer.jdbc.AllowCharacterTruncation=true

Aby włączyć możliwość ustawiania maksymalnej wartości dodatniej lub ujemnej pola liczbowego, gdy podane pole liczbowe przekracza zdefiniowaną granicę, należy ustawić następującą właściwość w pliku AcsConfig.properties:
    com.ibm.iaccess.dataxfer.jdbc.AllowNumericOverflow=true

Gdy te właściwości są ustawione na wartość true, żądanie przesyłania będzie kontynuowane bez generowania żadnych wskazań o wystąpieniu obcięcia lub przepełnienia. Pola znakowe będą obcinane na końcu. Pola liczbowe będą ustawiane na wartość maksymalną w przypadku przepełnienia i na wartość minimalną w przypadku niedomiaru.

Innym rozwiązaniem jest ustawianie powyższych właściwości z poziomu wiersza komend, podobnie jak innych właściwości Java:
    -Dcom.ibm.iaccess.dataxfer.jdbc.AllowCharacterTruncation=true
    -Dcom.ibm.iaccess.dataxfer.jdbc.AllowNumericOverflow=true

9.8.3 Nazwa arkusza funkcji przesyłania danych

Podczas pobierania nazwa arkusza jest generowana w oparciu o nazwę biblioteki źródłowej systemu IBM i oraz pliku z dodanym ciągiem znaków „>Sheet#”, gdzie symbol # zostaje zastąpiony odpowiednim numerem arkusza. Na przykład:
    qiws.qcustcdt>Sheet1

W celu przesłonięcia części nazwy arkusza odpowiadającej bibliotece i plikowi można określić następującą właściwość:
    com.ibm.iaccess.dataxfer.SheetId=łańcuch
Spowoduje to wygenerowanie nazwy:
    łańcuch>Sheet1

  10. Po wprowadzeniu poprawnych referencji zostanie wyświetlona konsola 5250.

http://pic.dhe.ibm.com/infocenter/iseries/v7r1m0/topic/rzamh/rzamhsrvtoolsrvr4dstdst.htm

W przypadku systemów IBM i, dla których istnieje już konfiguracja konsoli LAN i które znajdują się już w używanej infrastrukturze sieciowej, wykonaj następujące kroki w celu znalezienia istniejących konfiguracji konsoli:

9.10 Dodatkowe czcionki

Do katalogu Fonts można dodać dodatkowe czcionki dla emulacji terminala 5250.

Aby udostępnić alternatywne położenie dla dodatkowych czcionek oraz/lub wiele położeń dla dodatkowych czcionek, należy ustawić właściwość com.ibm.iaccess.Fonts znajdującą się w pliku AcsConfig.properties.

Domyślny katalog Fonts można przesłonić, ustawiając następującą wartość właściwości com.ibm.iaccess.Fonts:

  2. <ścieżka>/<nazwa_pliku> konkretnego pliku czcionek (na przykład .ttf)
  3. Adres URL do pliku czcionki
  4. Rozdzielona średnikami lista dowolnych z powyższych elementów

Wartość <ścieżka> może być pełną ścieżką lub ścieżką względną rozpoczynającą się od położenia pliku acsbundle.jar. W emulacji terminala 5250 rozpoznawane będą tylko czcionki o stałej szerokości.

     com.ibm.iaccess.Fonts=/Users/All/Fonts;Fonts

9.12 Zintegrowany system plików (Integrated File System - IFS)

9.12.1 IFS, QFileSvr.400 i bezpieczeństwo

Zintegrowany system plików używa systemu plików QFileSvr.400 do kopiowania lub wysyłania obiektów z jednej partycji systemu IBM i do innej. Oznacza to, że profil użytkownika i hasło w obu partycjach muszą być zgodne, a partycje muszą mieć tę samą wartość systemową poziomu hasła (QPWDLVL). W przypadku korzystania z uwierzytelniania Kerberos obie partycje muszą mieć skonfigurowaną Usługę uwierzytelniania sieciowego oraz Odwzorowanie tożsamości w przedsiębiorstwie (EIM). Więcej szczegółów zawiera temat „Security and object authority in the QFileSvr.400 file system” (Bezpieczeństwo i uprawnienia obiektów w systemie plików QFileSvr.400) Centrum Wiedzy IBM i.

Połączenia z innymi partycjami systemu IBM i nawiązywane przez system plików QFileSvr.400 nie są bezpieczne. Jeśli podczas kopiowania lub wysyłania obiektów między partycjami systemu IBM i wymagane jest bezpieczne połączenie, w konfiguracji partycji należy uwzględnić używanie protokołu SSL, w obszarze Konfiguracje systemu musi być dla danej partycji włączona opcja Użyj protokołu SSL do połączenia, a w pliku AcsConfig.properties musi być ustawiona następująca właściwość:
    com.ibm.iaccess.ifs.transferMechanism=ACS
Innym rozwiązaniem jest ustawianie tej właściwości z poziomu wiersza komend, podobnie jak innych właściwości Java:
    -Dcom.ibm.iaccess.ifs.transferMechanism=ACS

9.12.2 Uprawnienia systemu plików IFS

Obsługa zintegrowanego systemu plików wymaga uprawnień do korzystania z poniższych komend CL:

    CPY
    CPYFRMSTMF
    CPYTOSTMF
    CRTDIR
    CRTLIB
    CRTSAVF
    RMVDIR
    RST
    SAV
    SETASPGRP

Jeśli użytkownik nie może korzystać z tych komend CL, działania Kopiuj, Wklej, Wyślij, Usuń i Nowy folder kończą się niepowodzeniem.

9.12.3 Ograniczenia systemu plików IFS

Obsługa zintegrowanego systemu plików nie zezwala na wykonywanie działań na obiektach, które posiadają ścieżki rozpoczynające się od łańcucha /QFileSvr.400, /QNTC lub /QOPT. Jeśli bieżąca ścieżka katalogu lub dowolnego zaznaczonego obiektu zaczyna się od łańcucha /QFileSvr.400, /QNTC lub /QOPT, wszystkie pozycje w menu Działania zostają wyłączone, a menu kontekstowe nie jest wyświetlane.

9.12.4 IFS i niezależne ASP

Obsługa zintegrowanego systemu plików nie zezwala na wykonanie działania Kopiuj, Wklej lub Wyślij w przypadku obiektu źródłowego lub ścieżki docelowej znajdujących się w niezależnej ASP.

9.13 Terminal SSH (Secure Shell)

W przypadku platform, na których domyślnie istnieje bezpieczna powłoka (Secure Shell) lub w których została ona zainstalowana, w głównym interfejsie GUI produktu IBM i Access Client Solutions zostanie wyświetlona opcja terminala SSH. Terminal SSH uruchomi emulator terminala do zintegrowanego systemu plików IBM i.

Większość systemów Mac, Linux i innych systemów pochodzących od systemu UNIX ma domyślnie SSH (Secure Shell). W przypadku tych platform zakłada się, że klient OpenSSH istnieje i że znajduje się w katalogu /usr/bin/ssh. Użytkownicy systemów Windows muszą zainstalować jedną z bezpiecznych powłok dostępnych dla systemu Windows.

Bezpieczne powłoki obsługiwane przez produkt IBM i Access Client Solutions:

         Linux:       Xterm, XterminalEmulator, MateTerminal, Terminator, GnomeTerminal, Konsole, Xfce4
         Mac:         Terminal, iTerm2
         Windows: cygwin, PuTTY, bash
                          cygwin wymaga zainstalowania pakietu openssh
                          bash jest dostępna jako część podsystemu Linux dla Windows

Jeśli zainstalowano wiele aplikacji SSH, można użyć właściwości com.ibm.iaccess.PreferredSSHClient w celu ustawienia preferowanego klienta SSH.

Przykłady dla systemu Linux:
         com.ibm.iaccess.PreferredSSHClient=XterminalEmulator
         com.ibm.iaccess.PreferredSSHClient=MateTerminal
         com.ibm.iaccess.PreferredSSHClient=Terminator
         com.ibm.iaccess.PreferredSSHClient=GnomeTerminal
         com.ibm.iaccess.PreferredSSHClient=Konsole
         com.ibm.iaccess.PreferredSSHClient=Xfce4

Przykłady dla systemu Mac:
         com.ibm.iaccess.PreferredSSHClient=iTerm2

Przykłady dla systemu Windows:
         com.ibm.iaccess.PreferredSSHClient=Cygwin
         com.ibm.iaccess.PreferredSSHClient=Putty
         com.ibm.iaccess.PreferredSSHClient=Bash

W przypadkach, gdy klient SSH nie jest zainstalowany w standardowym położeniu, można użyć właściwości com.ibm.iaccess.PreferredSSHClient w celu określenia pełnej ścieżki do zainstalowanego klienta SSH.

         com.ibm.iaccess.PreferredSSHClient=C:\PuTTY\putty.exe

Jeśli zostanie określona pełna ścieżka, może być także konieczne ustawienie właściwości com.ibm.iaccess.SSHClientOpt w celu udostępnienia wszystkich niezbędnych argumentów. Aby na przykład wyłączyć ostrzeżenia, jeśli nie jest używane przekazywanie X11:

         com.ibm.iaccess.SSHClientOpts=-x

9.13.1 Wymagania wstępne dotyczące terminala SSH

Należy zainstalować produkt 5733-SC1 (IBM Portable Utilities for i) w systemie IBM i.
Należy uruchomić demon SSH za pomocą następującej komendy:
         STRTCPSVR *SSHD

Jeśli w uruchomionym kliencie SSH zostanie odebrany błąd połączenia (np. odmowa połączenia) lub jeśli zostanie wyświetlone okno, ale natychmiast zniknie, może być konieczne uruchomienie demona SSH za pomocą powyższej komendy.

      Należy spróbować ustawić zmienną środowiskową JAVA_HOME jako systemową
      zmienną środowiskową danego systemu operacyjnego. W systemie operacyjnym systemowe
      zmienne środowiskowe mogą być nazywane globalnymi zmiennymi środowiskowymi.
      Po ustawieniu zmiennej
      środowiskowej JAVA_HOME należy zamknąć i ponownie otworzyć menedżer plików
      i ponownie spróbować kliknąć dwukrotnie plik binarny.
