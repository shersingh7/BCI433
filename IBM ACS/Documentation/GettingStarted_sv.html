1.0 Introduktion

IBM i Access Client Solutions är den senaste medlemmen i IBM i Access-produktfamiljen. Den innehåller en Java-baserad oberoende plattform som kan köras i de flesta operativsystem som har funktioner för Java inklusive Linux, Mac och Windows. IBM i Access Client Solutions innehåller de mest vanliga rutinerna för hantering av IBM i.

IBM i Access Client Solutions använder samma IBM i-värdservrar som andra produkter i IBM i Access-familjen och kräver samma IBM i Access-familjelicens (XW1) för 5250-emulering och dataöverföring.

Det finns även två valfria paket tillgängliga som innehåller mellanprogramvara för användning och utveckling av klienttillämpningar för Windows och Linux:
    IBM i Access Client Solutions - Windows Application Package
    IBM i Access Client Solutions - Linux Application Package

Kunder som använder IBM i 7.1 eller senare och har aktuella rättigheter kan skaffa IBM i Access Client Solutions på något av följande sätt:

  • Hämta IBM i Access Client Solutions-baspaketet.
  • Hämta baspaketet och valfria Windows- och Linux-programpaket från webbplatsen Entitled Software Support (ESS) under 5761-SS1 eller 5770-SS1.

De valfria Windows- och Linux-programpaketen är endast tillgängliga från webbplatsen Entitled Software Support (ESS).

Kunder kan anskaffa medier genom att beställa 5761-SS1 eller 5770-SS1 uppdateringsfunktion 6288. De fysiska medierna innehåller en körbar version av produkten vilket gör att du kan köra produkten direkt från cd-skivan. Medierna innehåller också ett zip-arkiv med produkten som kan kopieras och extraheras till valfri plats. Fysiska medier för IBM i Access Client Solutions innehåller inte de valfria tillämpningspaketen för Windows och Linux.

Se även:
    http://www.ibm.com/systems/power/software/i/access/solutions.html

Den senaste informationen om IBM i Access-produkterna finns på:
    http://www.ibm.com/systems/i/software/access/caann.html

2.0 Funktioner

Funktionerna i IBM i Access Client Solutions inkluderar:

  • Komplett 5250-sessionsemulering baserat på IBM Rational Host-On-Demand. Förutom alla 5250-skärmsessionsfunktioner i IBM i Access for Windows finns även följande funktioner:
    • tabbfunktion som gör att flera sessioner kan dela ett enskilt visningsområde
    • mushjulsfunktion för intuitiv bläddring mellan sidor
    • växling mellan språk utan omstart av datorn
    • samtidiga skärmsessioner med olika värdteckentabeller för olika språk i olika emuleringssessioner
  • Skrivaremulering
  • En 5250-sessionshanterare med modell efter sessionshanteraren i IBM Personal Communications som används för hantering av alla 5250- sessioner.
  • Dataöverföring för överföring av data från/till IBM i-databasen till/från olika filtyper på datorn, t.ex. OpenDocument-kalkylark (*.ods), Excel-arbetsböcker (*.xlsx) och andra filformat.
  • Utskrifter är ett gränssnitt där du kan se filer i IBM i-utköer och även hämta filerna till klientsystemet.
  • En virtuell kontrollpanel med ett grafiskt gränssnitt till IBM i Operations Console.
  • 5250-emulering för LAN-, HMC- och FSM-konsoler.
  • Konsolidering för HMI-konfigurationer, inklusive ASMI, IVM och HMC.
  • Startfunktion för IBM Navigator for i med standardwebbläsare.
  • Databasfunktioner som Kör SQL-skript och SQL Performance Center.

Windows-programpaketet innehåller:

  • Uppkoppling till Db2 for i med ODBC, .Net och OLE DB
  • Programming Toolkit för åtkomst till IBM i-systemobjekt
  • Support för TLS/SSL-uppkopplingar
  • AFP-skrivardrivrutin

Linux-programpaketet innehåller:

  • En ODBC-drivrutin för Db2 for i och support för 64-bitars ODBC-datatyper.

5.0 Installation

Paketeringen av IBM i Access Client Solutions ger flera installationsalternativ från väldigt enkel användarinstallation till mer avancerad fleranvändarinstallation.

Installationsskripten finns i mapparna Mac_Application, Linux_Application och Windows_Application (se avsnittet 4.0 Produktinnehåll) som kan användas för olika installationsalternativ.

Information om installation för en enskild användare på Mac, Linux och Windows eller en installation för flera användare på en Windows-dator finns i QuickStartGuide. QuickStartGuide förklarar hur installationsskripten ska användas för de här installationstyperna.

Om du är administratör och planerar att driftsätta för flera användare läser du artikeln IBM i Access Client Solutions: Customization and deployment made easy. Artikeln förklarar hur en administratör kan anpassa installationen för flera användare före driftsättning genom att använda alternativet /AdminConfig.

För administratörer som vill automatisera installationen i bakgrundläge används parametern /AdminConfig för att förkonfigurera installationen. Sedan används parametern /Q vid själva installationen. Exempel:

    Windows_Application\install_acs_64.js /AdminConfig
    Windows_Application\install_acs_64.js /Q

Det går även att packa upp zip-filen på valfri plats. Det kan vara någon plats där datorn har läsbehörighet för filerna. Det innefattar den lokala hårddisken, en fjärrenhet (delat nätverk) eller ett flyttbart lagringsmedium, t.ex. en cd-skiva eller en USB-enhet. När du packar upp zip-filen slutförs installationen.

Teknisk anmärkning:
Visa arkivfunktioner bevarar inte alla sparade filattribut. För Mac och Linux är kommandot unzip vanligen ett bättre val än kommandot jar. Det finns mer information i avsnitt 6.0 Filbehörigheter.

5.1 Uppdatera en befintlig installation

Förbättringar och fixar kan göras tillgängliga på periodisk basis. Uppdateringarna tillhandahålls som en komplett produktinstallation. När sådana uppdateringar blir tillgängliga ska du uppdatera den befintliga installationen.

Uppdatera en befintlig installation som installerades med installationsskripten genom att extrahera innehållet för en senare version av produkten. Anropa installationsskriptet från den nya versionen på samma sätt som under den första installationen. Då uppdateras produktfilerna utan att ändra användarnas konfiguration.

För administratörer som underhåller en version av produkten på en central plats som accessas av många användare går det att extrahera innehållet för den senare versionen ovanpå den befintliga. Om du har gjort anpassningar och ändrat filen AcsConfig.properties ska du spara filen innan du extraherar innehållet för den nya. När du har extraherat innehållet och återställt AcsConfig.properties kan användarna köra installationsskriptet på samma sätt som för den första installationen.

Om du väljer att installera produkten utan att använda installationsskripten måste du extrahera innehållet för den senare versionen ovanpå den befintliga. Du kan behöva spara och återställa filen AcsConfig.properties om du har ändrat den.

6.0 Filbehörigheter

Avsnitt 7.0 Starta produkten beskriver olika sätt att starta IBM i Access Client Solutions. Om du ska använda en av de tillhandahållna binära filerna eller skripten för att starta produkten måste du se till att det finns behörighet att köra filerna eller skripten. Vilka filbehörigheter som tilldelas när zip-arkivet packas upp bestäms av flera faktorer, t.ex. operativsystem, vilket program som användes för att packa upp zip-arkivet, användarbehörighet osv.

Kontrollera filbehörigheterna om det inte går att använda något av de binära filerna eller skripten. I följande avsnitt beskrivs några sätt att kontrollera filbehörigheter.

6.1 Filbehörigheter (Linux, Mac och AIX)

För UNIX-liknande operativsystem kan du använda följande kommando från ett skal eller terminalfönster för att kontrollera behörigheterna för en fil:

        ls -l <fil>

Om du vill ändra behörigheterna för en fil kan du använda följande kommando:

    chmod <behörighet> <fil>

Exempel:
Lägga till läs- och körbehörighet för alla: 
    chmod a+rx <fil>
Ge filägaren läs-, skriv- och körbehörighet men endast läs- och körbehörighet för alla andra: 
    chmod 755 <fil>

Ytterligare information om kommandona ls och chmod finns på Internet.

6.2 Filbehörigheter (Windows)

I Windows högerklickar du på filen i Utforskaren och väljer Egenskaper. Filbehörigheterna visas på fliken Säkerhet. Kontrollera att du har läs- och körbehörighet.

I senare versioner av Windows kan du även använda kommandot icacls för att se och ändra filbehörigheter.

7.0 Starta produkten

Det finns flera sätt att starta IBM i Access Client Solutions. Det lättaste sättet att starta produkten om installationsskripten används för att installera produkten beskrivs i QuickStartGuide.

Resten av det här dokumentet beskriver alternativa sätt att starta produkten om den installerades utan att använda skripten. Eftersom det finns en mängd sätt och platser för hur och var Java kan installeras kan vissa metoder kräva ytterligare konfigurering. Försök med en annan metod om någon av metoderna nedan inte fungerar. I vissa fall finns ytterligare vägledning. Avsnitt 11.0 Vanliga frågor kan också vara till hjälp.

Om du använder en binär fil eller ett skript måste de finnas i samma katalogstruktur som i zip-arkivet. Du kan även kopiera eller flytta den binära filen eller skriptet för plattformen till samma katalog som acsbundle.jar finns i.

7.1 Starta produkten

Om du vill starta produkten från en filvisare (t.ex. Windows Utforskaren, Mac OS X Finder etc.) genom att använda en plattformsspecifik binär fil, letar du reda på underkatalogen i Start_Programs som identifierar operativsystemet och maskinvaruarkitekturen.

Gå till den binära fil som gäller för operativsystemet. Dubbelklicka på filen för att starta produkten. Du kan även starta produkten med den binära filen från en kommandoprompt, terminal eller shell.

Om du får följande fel:
    "Error loading Java module."
kunde IBM i Access Client Solutions inte hitta någon Java-installation på en förväntad plats. Du kan försöka med hjälp av anvisningarna i något av följande avsnitt:
    7.1.1 Starta produkten - Ytterligare alternativ
    7.2 Starta produkten med ett skript
    7.3 Starta produkten från kommandoraden

Teknisk anmärkning för Mac:
Om du blir ombedd att installera "Java 6 SE runtime" behöver du inte installera Java 6. Installera Java 8 och försök starta produkten igen. Om du använder acslaunch_mac för att starta produkten och meddelandet om att installera "Java 6 SE runtime" fortsätter att visas går du till Utilities och startar en terminalsession. Kör följande kommandon och försök sedan starta produkten igen: 

    sudo mkdir -p /System/Library/Java/JavaVirtualMachines/1.6.0.jdk
    sudo mkdir -p /System/Library/Java/Support/Deploy.bundle

7.1.1 Starta produkten - Ytterligare alternativ

Du kan även försöka med någon av följande metoder när du använder den binära filen för plattformen. Metoderna identifierar vilken JRE (Java Runtime Environment) som ska användas för att starta produkten. Information om hur du hittar Java Home-sökvägen på datorn finns i avsnitt 7.1.2 Hitta Java Home-sökvägen. Metoderna kan endast användas på Linux- och Windows-plattformar:

  • Ange miljövariabeln JAVA_HOME till Java Home-sökvägen (eller)
  • Använd alternativet -vm för det binära kommandot för att ange Java Home-sökvägen. Om du vill ha mer information använder du alternativet -h för det binära kommandot. (eller)
  • Kopiera JRE-katalogstrukturen till samma katalog som den binära filen finns i.

7.1.2 Hitta Java Home-sökvägen

Om du kan starta produkten med en av metoderna i avsnitt
    7.2 Starta produkten med ett skript eller
    7.3 Starta produkten från kommandoraden
kan du bestämma Java Home-sökvägen på datorn från huvudgränssnittet i IBM i Access Client Solutions. Välj följande från menyraden:
    Hjälp -> Om
    Sökvägen till java.home visas i det här fönstret.

Egenskapen java.home innehåller platsen för Java Home-katalogen på datorn. Det är den sökvägen du ska ange för miljövariabeln JAVA_HOME eller när du använder kommandoalternativet -vm.

7.1.2.1 Hitta Java Home-sökvägen (Windows)

Sök efter java.exe på Windows-plattformar. Filen finns vanligen i en bin-katalog eller jre/bin-underkatalog i Java Home-sökvägen. Java Home-sökvägen kan användas för att ange miljövariabeln JAVA_HOME eller för kommandoalternativet -vm.

7.1.2.2 Hitta Java Home-sökvägen (Linux)

I Linux kan du använda kommandot which:

    which java
Då visas sökvägen till java-kommandot eller en symbolisk länk till det.

Lös de symboliska länkarna tills du kommer till den binära filen för java-kommandot. Du kan lösa symboliska länkar med kommandoalternativet -l:

    ls -l <fil>
Filen finns vanligen i en bin-katalog eller jre/bin-underkatalog i Java Home-sökvägen. Java Home-sökvägen kan användas för att ange miljövariabeln JAVA_HOME eller för kommandoalternativet -vm.

7.2 Starta produkten med ett skript

Från och med v1r1m5 har katalogen Start_Scripts bytt namn till Sample_Scripts och finns i katalogen Documentation.

Det rekommenderade sättet att starta produkten är att använda en plattformsspecifik binär fil som finns i Start_Programs. Skripten i Sample_Scripts ska endast användas om de plattformsspecifika binära filerna inte fungerar.

Starta produkten från en filsystembläddrare (t.ex. Windows Utforskaren, Mac OS X Finder osv.) med ett av skripten, gå till skriptet i katalogen Sample_Scripts för operativsystemet.

De flesta andra än Windows-baserade operativsystem har perl tillgängligt som standard. Katalogen Sample_Scripts/Linux_Mac_Other innehåller ett perl-skript (med tre olika filtillägg) som kan användas för att starta produkten på valfri plattform där perl är tillgängligt. Välj den fil som har en filtyp som operativsystemet känner igen som ett perl-skript.

Windows-baserade operativsystem har JScript tillgängligt som standard. Katalogen Sample_Script/Windows innehåller ett JScript som kan användas för att starta produkten i Windows-operativsystem.

Leta efter det skript som operativsystemet känner igen med någon plattformsspecifik metod (t.ex. Utforskaren i Windows, Mac OS X Finder, osv.). Dubbelklicka på skriptet för att starta produkten. Du kan även starta produkten med skriptet från en kommandoprompt, terminal eller shell.

7.3 Starta produkten från kommandoraden

Du kan även starta produkten från en kommandoprompt, terminal eller shell.

    java -Xmx1024m -jar <sökväg>/acsbundle.jar
<sökväg> är platsen för produktens exekverbara jar-fil

Exempel:
    java -Xmx1024m -jar V:/plats/acsbundle.jar
eller 
    <java-sökväg>java -jar V:/plats/acsbundle.jar
<java-sökväg> är platsen för java-kommandot för JDK 6.0 eller senare.

Information om hur du tar reda på den fullständiga sökvägen till java-kommandot finns i avsnitt 7.1.2 Hitta Java Home-sökvägen.

Du kan även använda något av programmen eller skripten från kommandoraden. Exempel:

    /Product/Location/Start_Programs/Linux_x86-64/acslaunch_linux-64
    /Product/Location/Start_Programs/Mac_i386-32_x86-64/acslaunch_mac
    C:\Product\Location\Start_Programs\Windows_x86-64\acslaunch_win-64.exe

Teknisk anmärkning:
På de flesta plattformar är standardvärdet för maximal JVM-högstorlek för lågt för att man ska kunna utnyttja flera funktioner i IBM i Access Client Solutions. Rekommenderad minimistorlek för maximal högstorlek är 1 gigabyte (-Xmx1024m). Ett värde lägre än 1 GB eller standardvärdet kan resultera i minnesundantagsfel.

8.0 Konfiguration

Lägg till en systemkonfiguration för varje IBM i-system som du ska använda eller hantera. Välj Systemkonfigurationer från rutinerna i Hantering. Klicka sedan på Ny. Ange systemnamnet på fliken Allmänt. Det är endast systemnamnet som behövs för att kunna utföra allmänna rutiner.

Klicka på OK för att spara informationen eller välj Spara/Ny om du vill lägga till fler systemkonfigurationer.

Du kan när som helst lägga till nya system eller uppdatera konfigurationer på flikarna Allmänt, Uppkoppling eller Konsol.

För konsolrutiner krävs ytterligare konfiguration. Konsol- konfigurationer kopplas automatiskt till det systemnamn du har angett på fliken Allmänt. Om du vill ange konsolkonfigurationen för ett system väljer du Systemkonfigurationer från rutinerna i hantering. Välj Ny eller Ändra. Klicka sedan på fliken Konsol. 5250-konsolrutinen kräver en konfigurerad LAN-konsol eller en konfigurerad HMC-konsol. Information om hur du konfigurerar en LAN- eller HMC-konsol finns i avsnitt 9.9 Upprätta en konsoluppkoppling till IBM i.

HMI-rutinen (Hardware Management Interface) kräver en konfigurerad HMI. Du kan ange upp till två HMI-konfigurationer.

Välj Stäng på systemkonfigurationspanelen när du är klar.

Välj ett system i listrutan System i IBM i Access Client Solutions- panelen. Alla konsolrutiner kopplar automatiskt valt system (på fliken Allmänt) till konsolkonfigurationen (som har angetts på fliken Konsol).

Nu kan du välja rutin för valt system. Om du väljer en konsolrutin som det inte finns någon motsvarande information för på fliken Konsol, visas ett felmeddelande.

9.1 Fler kommandoradsalternativ

Många av de funktioner som är tillgängliga från huvudgränssnittet är även tillgängliga från kommandoraden. Funktionerna kan anropas med hjälp av parametrar till alla kommandoradsalternativ som visas i:
avsnitt 7.3 Starta produkten från kommandoraden

Exempel: 

    Start_Programs\Windows_x86-64\acslaunch_win-64.exe parameter1 parameter2 ...

I följande avsnitt visas endast de ytterligare parametrarna.

9.1.1 Backup

/PLUGIN=backup  [/file=<filnamn>]
    <filnamn> är namnet på den fil som ska skapas

Konfigurationen sparas i angiven fil. Filen kan användas som indata till kommandoalternativet restore på samma eller någon annan dator (oavsett operativsystem).

Platsen där konfigurationen sparas bestäms av egenskapen

    com.ibm.iaccess.AcsBaseDirectory
som finns i filen AcsConfig.properties.

Funktionen motsvarar Arkiv->Exportera konfiguration i huvudgränssnittet.

9.1.2 Återskapa

/PLUGIN=restore /file=<filnamn>
    <filnamn> är platsen för filen som skapades av Backup
En sparad konfiguration återställs från angiven fil. Alla befintliga konfigurationer som inte finns i angiven fil tas bort.

Platsen för den återställda konfigurationen bestäms av egenskapen 

    com.ibm.iaccess.AcsBaseDirectory
som finns i filen AcsConfig.properties.

Funktionen motsvarar Arkiv->Importera konfiguration från huvudgränssnittet.

9.1.3 Certdl

/PLUGIN=certdl  /SYSTEM=<system>
Hämtar en certifikatutfärdare från angivet IBM i-system och lagrar det i användarens lokala lager för betrodda certifikat. Det krävs för serverautentisering med SSL.

9.1.4 Cfg

/PLUGIN=cfg     /LIST

                /LIST   - listar konfigurerade system med deras uppkopplingsalternativ


/PLUGIN=cfg     /SYSTEM=<system> [/ipaddr=<frekvens>] [/userid=<användar-ID>]
                                 [/ssl=<switch>]
                                 [/5250path=<sökväg>]
                                 [/del]  [/r]

                /SYSTEM  - namnet på systemet
                /ipaddr  - när en uppkoppling begärs bestämmer det här värdet om det
                  ska göras en sökning efter IP-adressen.                 Giltiga frekvenser:
                      ALWAYS - sök efter IP-adress för varje uppkoppling
                      HOURLY, DAILY, WEEKLY - sök efter IP-adress om tidsperioden
                      har passerat sedan den senaste sökningen
                      IP-adress - Om en IP-adress anges förmodas frekvensen för
                      sökning vara NEVER
               /userid   - en användares användar-ID kan också anges till följande värden:
                      *SHARE        - frågar en gång efter inloggningskreditiv som
                                      ska delas av system med det här alternativet
                      *PROMPTALWAYS - frågan minst en gång för varje uppkoppling
                      *KERBEROS - Kerberos-huvudnamn, ingen bekräftelse
               /ssl      - 0 avaktiverar SSL-läge, 1 aktiverar SSL-läge
               /5250path - sökvägen till 5250-emuleringsprofilerna
                  /5250path kan sättas genom att välja Arkiv->Ändringshistorik... från
5250-sessionshantering
                  /del   - tar bort befintlig konfiguration
                  /r     - ersätter befintlig konfiguration
Det gör att olika konfigurationsalternativ kan anges på kommandoraden. Alternativen kan även anges i huvudgränssnittet under Systemkonfigurationer.

9.1.11 Ping

/PLUGIN=ping /SYSTEM=<system> [</alternativ>]
    Giltiga alternativ:
        /SSL=<1|0>        Aktivera/avaktivera SSL
        /ACCEPTALLCERTS=<1|0> Om alla SSL-certifikat ska läggas till i den betrodda uppsättningen (om SSL används).
        /SERVERAUTH=<1|0> Aktivera/avaktivera SSL-serverautentisering
                          (standard är avaktivera). Alternativet ignoreras om
                          ingen SSL-kontroll sker.
        /GUI=<1|0>        Växlar gränssnittsfönstret av/på (standard är av vid start
                          från kommandoraden)
        /PORTS=<port1,port2> En kommaseparerad lista över portar som ska testas.
                                   Det kan vara nummer eller tjänstenamn (t.ex. /PORTS=as-signon,as-sts).
                                   Om inget värde anges testas en standarduppsättning portar.
                                   Med .CONSOLE kontrolleras en lista över konsolspecifika portar.

        /TIMEOUT=<seconds>   Timeoutvärde i sekunder.
Insticksprogrammet kontrollerar uppkopplingen till IBM i genom att upprätta en uppkoppling till lämplig port. Om det är en SSL- uppkoppling görs ett försök till handskakning. Om anropet sker från huvudgränssnittet eller med /GUI=1, öppnas en dialogruta. Om anropet sker från kommandoraden, utan /GUI=1, skickas utdata till konsolen.

Som standard markeras följande tjänster:
    as-central, as-rmtcmd, as-database, as-dtaq, as-file, as-netprt, drda och
    as-signon

Funktionen kan startas från huvudgränssnittet med:
    Systemkonfigurationer
    Markera ett system och välj Redigera
    Fliken Allmänt
    Verifiera uppkoppling

9.2 Filassociationer

Vissa konfigurationsfiler som genereras av IBM i Access Client Solutions kan köras från kommandoraden om de ges som den första och enda parametern. När dessa filer med specifika filtyper ges som den första parametern kopplar IBM i Access Client Solutions filen med den funktion som ska anropas och tillhandahåller filen som indata till den funktionen.

Följande filtillägg kan associeras: 

    .dttx   - överföringsbegäran i Dataöverföring
    .dtfx   - hämtningsbegäran i Dataöverföring
    .hod    - sessionsprofil för 5250-emulering
    .bchx   - flersessionsprofil för 5250-emulering
    .ws     - sessionsprofil för 5250-emulering (Personal Communications)
    .bch    - flersessionsprofil för 5250-emulering (Personal Communications)

9.3 Ändra konfigurationsplats

Som standard har varje användare en egen unik konfigurationsplats. Konfigurationsplatsen kan bytas genom att ange egenskapen
    com.ibm.iaccess.AcsBaseDirectory
Den här egenskapen finns i filen AcsConfig.properties.

AcsConfig.properties finns på två platser när produkten levereras. Den finns i filen acsbundle.jar. Den finns även i produktens zip-arkivfil och i samma katalog som filen acsbundle.jar när zip-filen packas upp.

Vid start används endast den första AcsConfig.properties-filen som hittas. Först kontrolleras katalogen där filen acsbundle.jar finns. Om AcsConfig.properties inte finns i samma katalog som acsbundle.jar används AcsConfig.properties i filen acsbundle.jar.

Du kan välja att uppdatera AcsConfig.properties i acsbundle.jar med en anpassad konfigurationssökväg. Om du gör det måste du kontrollera att katalogen där acsbundle.jar finns inte innehåller filen AcsConfig.properties. Om den gör det används den filen i stället. Det ger möjlighet att kunna distribuera konfigurationsplatsen med filen acsbundle.jar och samtidigt kunna åsidosätta den.

Det finns speciella nyckelord som kan användas vid definition av konfigurationssökvägen. När nyckelorden används i den angivna sökvägen, ersätts de med den text eller sökväg de definierar. Endast ett nyckelord kan användas i konfigurationssökvägen. Speciella nyckelord och betydelse:

	{USER} - Aktuellt användar-ID. Nyckelordet kan finnas var som helst i sökvägen.

Följande nyckelord kan endast finnas i början av angiven sökväg: 

	{PRODUCTDIR} - sökvägen till samma katalog som acsbundle.jar
	{TEMPDIR}    - sökvägen till en plattformsspecifik tempkatalog
	{ROOT}       - sökvägen till filsystemets rot
	{HOME}       - sökvägen till användarens home-katalog
	{DEFAULT}    - standardsökvägen för produkten

Teknisk anmärkning:
Det rekommenderas inte att dela en konfiguration mellan flera användare. Exempel: Om X är en delad nätverksenhet kan följande inställning ge oväntade resultat: 

    com.ibm.iaccess.AcsBaseDirectory=X:/Shared_Network_drive/config_directory

Det finns flera problem med att flera användare delar den här konfigurationssökvägen:

  1. Oväntade resultat inträffar om flera användare samtidigt använder konfigurationen.
  2. I exemplet förutsätts att varje användare har X mappad till samma plats.
  3. En nätverksenhet i konfigurationen fungerar inte från en Linux- eller Mac-klient. Med nyckelorden i exemplet nedan fungerar det i Windows, Linux och Mac.

Om flera användare ska dela en konfigurationssökväg ska nyckelordet {USER} användas för att undvika kollision med andra användare. Det ersätts med den aktuella användarens användar-ID.

Använd ett snedstreck (/) i stället för omvänt snedstreck (\) som katalogavgränsare i konfigurationssökvägen. Det fungerar i alla operativsystem inklusive Windows.

Rekommenderade exempelkonfigurationer:

Exempel 1 - lokal konfiguration för aktuell användare (standard): 

    com.ibm.iaccess.AcsBaseDirectory=
Om AcsBaseDirectory inte anges sparas konfigurationen för användaren i en plattformsberoende sökväg. Det är standardinställningen för IBM i Access Client Solutions.

Exempel 2 - unik fjärrkonfiguration (eller lokal) för varje användare: 

    com.ibm.iaccess.AcsBaseDirectory={ROOT}/config_directory/{USER}/
Fjärrkonfiguration (eller lokal) baserat på platsen för {ROOT}.

Exempel 3 - unik fjärrkonfiguration (eller lokal) för varje användare: 

    com.ibm.iaccess.AcsBaseDirectory={PRODUCTDIR}/config_directory/{USER}/
Fjärrkonfiguration (eller lokal) baserat på platsen för {PRODUCTDIR}.

Exempel 4 - lokal konfiguration på flyttbara medier (t.ex. USB): 

    com.ibm.iaccess.AcsBaseDirectory={PRODUCTDIR}/config_directory
Eftersom sökvägen till det flyttbara mediet varierar beroende på vilket system som används kan konfigurationen vara relativ till produktfilerna. I det här exemplet anges inte nyckelordet {USER} vilket gör att konfigurationen på USB-enheten kan användas av den som använder USB-enheten, oavsett användar-ID.

9.4.1 Importera konfigurationsinställningar automatiskt

Följande egenskaper i AcsConfig.properties kan användas för att automatiskt konfigurera inställningar för nya användare eller för att uppdatera konfigurationen för befintliga användare:

   com.ibm.iaccess.autoimport
   com.ibm.iaccess.autoimport.version

Steg som behövs:

  1. Skapa den konfiguration som ska spridas till en eller fler användare.
  2. Exportera konfigurationen till en fil genom att välja Arkiv -> Exportera konfiguration i huvudgränssnittet eller använd kommandoradsalternativet i avsnitt 9.1.1 Backup.
  3. Flytta konfigurationsfilen till önskad plats. Det finns flera alternativ som beskrivs senare.
  4. Ange egenskapen com.ibm.iaccess.autoimport i AcsConfig.properties till sökvägen för konfigurationsfilen.
  5. Ange com.ibm.iaccess.autoimport.version till ett heltalsvärde som motsvarar versionen för konfigurationsfilen.

Så här fungerar det:
Den sparade konfigurationsfilen som refereras av egenskapen com.ibm.iaccess.autoimport importeras automatiskt när heltalsvärdet för egenskapen com.ibm.iaccess.autoimport.version inte stämmer med det senaste importvärdet. Förutom en möjlighet att definiera en initial konfiguration och tillhandahålla uppdateringar till befintliga konfigurationer är det även ett sätt att återställa en konfiguration. Konfigurationen uppdateras när heltalsvärdet för versionsegenskapen inte stämmer med den senaste versionen som importerades. En importerad konfiguration kan ändra konfigurationen för ett befintligt system i en användares konfiguration, men ett system tas aldrig bort från konfigurationen.

Sökvägen till egenskapen com.ibm.iaccess.autoimport kan vara absolut, en URL eller ha angetts med nyckelord som har definierats i avsnitt 9.3. Exempel: 

    com.ibm.iaccess.autoimport=C:/acs_bak.zip

    com.ibm.iaccess.autoimport=file///C:/acs_bak.zip

    com.ibm.iaccess.autoimport=http://företag.com/sökväg/fil/acs_bak.zip

    com.ibm.iaccess.autoimport=ftp://företag.com/sökväg/fil/acs_bak.zip

    com.ibm.iaccess.autoimport={PRODUCTDIR}/acs_bak.zip

Ytterligare flexibilitet fås genom att konfigurationsfilen kan distribueras inom acsbundle.jar eller i samma katalog som acsbundle.jar. I båda fallen anges com.ibm.iaccess.autoimport med samma namn som filen utan inledande sökväg:

    com.ibm.iaccess.autoimport=acs_bak.zip

Specialvärdet * får användas för com.ibm.iaccess.autoimport.version: 

    com.ibm.iaccess.autoimport.version=*

Då importeras konfigurationen oavsett tidigare version.

9.8 Dataöverföring

9.8.1 Dataöverföringsfunktioner för Excel- och Calc-kalkylark

Förutom funktioner för att hämta till en fil har Dataöverföring även funktioner för att hämta till ett aktivt Microsoft Excel- eller Open Office Calc-kalkylark. Det finns ett val för utdataenhet i dataöverföringsgränssnittet. Standardvärdet för utdataenhet är Fil. Om plattformen har funktioner för interaktion med Excel- och/eller Calc-kalkylark kan du välja Aktivt Excel-kalkylark eller Aktivt Calc- kalkylark i listrutan Utdataenhet.

Begränsningar:

  • Minst OpenOffice 3.4.1 krävs för Windows.
  • Minst OpenOffice 4.2.1 krävs för Linux och Mac och måste installeras på standardplatsen.
  • IBM i Access Client Solutions som körs på Linux eller Mac stöder endast interaktion med OpenOffice.
    IBM i Access Client Solutions som körs i Windows stöder interaktion med Microsoft Excel eller OpenOffice Calc.
  • Antalet bitar i OpenOffice måste motsvara antalet bitar i IBM i Access Client Solutions.
    Förklaring: OpenOffice är tillgänglig i både 32-bitars och 64-bitars versioner. Det går att köra båda versionerna på en dator med 64-bitars operativsystem. Vid interaktion med ett OpenOffice-kalkylark måste antalet bitar i OpenOffice och i IBM i Access Client Solutions vara samma, annars kan inte dataöverföringen interagera med kalkylarket. Antalet bitar i IBM i Access Client Solutions bestäms av den Java Virtual Machine (JVM) där den körs. Om antalet bitar inte är samma i OpenOffice och JVM som kör IBM i Access Client Solutions och du väljer ett aktivt kalkylark för utdataenheten visas ett felmeddelande som "Aktivt kalkylark hittades inte". Om du kör på en 64-bitarsdator kan du starta IBM i Access Client Solutions med den 32- eller 64-bitars binära startfilen för plattformen. Om stödet för det aktiva kalkylarket ska fungera, måste du starta IBM i Access Client Solutions med den plattformsspecifika binära kod som matchar bitantalet i den OpenOffice-tillämpning som används.
    Eftersom Mac endast hanterar JVM 64 bitar måste du installera 64-bitarsversionen av OpenOffice om du ska använda aktiva kalkylark i Mac.

9.8.2 Dataöverföringsfunktioner för teckentrunkering och numeriskt spill

Om ett tecken- eller numeriskt fält överstiger definierad längd under en dataöverföring stoppas överföringen.

Aktivera trunkering från slutet av teckenfält genom att definiera följande egenskap i filen AcsConfig.properties:
    com.ibm.iaccess.dataxfer.jdbc.AllowCharacterTruncation=true

Aktivera maximalt positivt eller negativt värde för numeriska fält när fältet överskrider den definierade gränsen med följande egenskap i filen AcsConfig.properties:
    com.ibm.iaccess.dataxfer.jdbc.AllowNumericOverflow=true

När egenskaperna har värdet true fortsätter överföringen utan indikering av trunkering eller spill. Teckenfält trunkeras i slutet. Numeriska fält sätts till det högsta värdet för spill och det minsta värdet för underspill.

Egenskaperna kan även sättas från kommandoraden som med andra java-egenskaper på följande sätt:
    -Dcom.ibm.iaccess.dataxfer.jdbc.AllowCharacterTruncation=true
    -Dcom.ibm.iaccess.dataxfer.jdbc.AllowNumericOverflow=true

9.8.3 Dataöverföring och arknamn

Det blir:

9.9 Upprätta en konsoluppkoppling till IBM i

Det krävs en 5250-konsol för att du ska kunna utföra administrativa uppgifter för IBM i. IBM i Access Client Solutions har funktioner för både LAN- och HMC- konsolkonfigurationer.

Om du känner till Servicevärdnamnet eller IP-adressen för IBM i eller värdnamnet eller IP-adressen till HMC-konsolen, kan du konfigurera konsolinformationen i IBM i Access Client Solutions med hjälp av följande steg:

  1. Starta IBM i Access Client Solutions på datorn.
  2. Klicka på Systemkonfigurationer på huvudpanelen.
  3. Klicka på knappen Ny och skapa en ny konfiguration eller klicka på Redigera och uppdatera en befintliga konfiguration.
  4. Markera fliken Konsol.
  5. Fyll i informationen för konsoltypen.

För ett IBM i-system där det inte redan finns en konsolkonfiguration (t.ex. ett nytt system) tilldelas automatiskt en IP-adress för konsol- uppkoppling under IPL. IP-adressen ligger inom intervallet 169.254.62.0 - 169.254.62.63. För dessa fall upprättas en konsoluppkoppling med IBM i Access Client Solutions på följande sätt:

  1. Avaktivera datorns funktioner för trådlös uppkoppling. Följande steg kräver en kabelansluten ethernetport.
  2. Koppla en ethernetkabel mellan datorn och T1-porten på IBM i-systemet.
  3. Starta IBM i Access Client Solutions på datorn.
  4. Klicka på Systemkonfigurationer på huvudpanelen.
  5. Klicka på knappen Hitta konsol från Systemkonfigurationer.

9.12 Integrerat filsystem (IFS)

9.12.1 IFS, QFileSvr.400 och säkerhet

IFS-support använder filsystemet QFileSvr.400 för att kopiera och skicka objekt från en IBM i-partition till en annan. Det betyder att användarprofil och lösenord på båda partitionerna måste vara samma samt att partitionerna måste ha samma systemvärde för lösenordsnivå (QPWDLVL). Om Kerberos-autentisering används måste båda partitionerna ha NAS (Network Authentication Service) och EIM (Enterprise Identity Mapping) konfigurerat. Det finns mer information i avsnittet "Security and object authority in the QFileSvr.400 file system" i IBM i Knowledge center.

Anslutningen till andra IBM i-partitioner via QFileSvr.400 är inte säker. Om en säker anslutning behövs vid kopiering och sändning av objekt från en IBM i-partition till en annan, måste partitionen konfigureras för SSL, systemkonfigurationer måste ha alternativet Använd SSL för anslutning valt för partitionen och följande egenskap måste sättas i filen AcsConfig.properties:
    com.ibm.iaccess.ifs.transferMechanism=ACS
Egenskapen kan även sättas från kommandoraden som med andra java-egenskaper på följande sätt:
    -Dcom.ibm.iaccess.ifs.transferMechanism=ACS

9.12.2 IFS-behörighet

IFS kräver behörighet för användning av följande CL-kommandon:

    CPY
    CPYFRMSTMF
    CPYTOSTMF
    CRTDIR
    CRTLIB
    CRTSAVF
    RMVDIR
    RST
    SAV
    SETASPGRP

Om användaren inte har behörighet för dessa CL-kommandon går det inte att kopiera, klistra in, skicka, ta bort eller skapa en ny mapp.

9.12.3 IFS-begränsningar

IFS tillåter inte åtgärder för objekt som har sökvägar som börjar med /QFileSvr.400, /QNTC och /QOPT. Om katalogsökvägen eller någon vald objektsökväg börjar med /QFileSvr.400, /QNTC eller /QOPT avaktiveras alla alternativ i menyn Åtgärder och kontextmenyn visas inte.

9.12.4 IFS och oberoende ASPer

IFS tillåter inte åtgärderna Kopiera, Klistra in och Skicka nät källobjektet eller målsökvägen finns i en oberoende ASP.

9.13 SSH-terminal (Secure Shell)

På plattformar där Secure Shell finns som standard eller har installerats, visas SSH-terminalfunktionen i huvudgränssnittet i IBM i Access Client Solutions. SSH-terminalen startar en terminalemulator till det IBM i-integrerade filsystemet.

De flesta Mac-, Linux- och andra UNIX-system har redan Secure Shell som standard. För dessa plattformar förutsätts att det finns en OpenSSH-klient i /usr/bin/ssh. Windows-användare måste installera tillgängligt Secure Shell för Windows.

Secure Shell som supportas av IBM i Access Client Solutions:

         Linux:       Xterm, XterminalEmulator, MateTerminal, Terminator, GnomeTerminal, Konsole, Xfce4
         Mac:         Terminal, iTerm2
         Windows: cygwin, PuTTY, bash
                          cygwin kräver att openssh-paketet är installerat
                          bash är tillgängligt som en del av Linux-delsystemet för Windows

Om det finns fler SSH-program installerade kan du använda egenskapen com.ibm.iaccess.PreferredSSHClient till att ange önskad SSH-klient.

Linux-exempel:
         com.ibm.iaccess.PreferredSSHClient=XterminalEmulator
         com.ibm.iaccess.PreferredSSHClient=MateTerminal
         com.ibm.iaccess.PreferredSSHClient=Terminator
         com.ibm.iaccess.PreferredSSHClient=GnomeTerminal
         com.ibm.iaccess.PreferredSSHClient=Konsole
         com.ibm.iaccess.PreferredSSHClient=Xfce4

Mac-exempel:
         com.ibm.iaccess.PreferredSSHClient=iTerm2

Windows-exempel:
         com.ibm.iaccess.PreferredSSHClient=Cygwin
         com.ibm.iaccess.PreferredSSHClient=Putty
         com.ibm.iaccess.PreferredSSHClient=Bash

Om SSH-klienten inte är installerad i standardkatalogen kan du använda egenskapen com.ibm.iaccess.PreferredSSHClient till att ange fullständig sökväg till den installerade SSH-klienten.

         com.ibm.iaccess.PreferredSSHClient=C:\PuTTY\putty.exe

Om den fullständiga sökvägen anges kan du även behöva ange com.ibm.iaccess.SSHClientOpt med alla argument som krävs. Exempel: Avaktivera varningar om X11 forwarding inte används:

         com.ibm.iaccess.SSHClientOpts=-x

9.13.1 SSH-terminal - förkrav

Installera 5733-SC1 IBM Portable Utilities for i på IBM i-systemet.
Starta Secure Shell Daemon med kommandot:
         STRTCPSVR *SSHD

Om det uppstår ett anslutningsfel för den startade SSH-klienten (t.ex. "connection refused"), eller om ett fönster visas och försvinner med detsamma, kan du behöva starta om SSH daemon med ovanstående kommando.
